# Per-prop usage taxonomy in component docs — Design Spec

## Context

Design agents (the consumer-side `/klp-design` pipeline in downstream apps) read each component's doc page to decide which props to pass. They hardcode interactive props that the component would auto-derive. Concrete example: Input's `state` prop is auto-derived from `disabled`, `aria-invalid`, focus, and value; passing `state="default"` explicitly freezes the UI and breaks focus/filled transitions.

Root cause: doc pages list every variant axis from the Figma spec as if it were a runtime prop, without distinguishing between:
- props the agent MUST pass (label, onClick, href…)
- props the agent MAY pass to override defaults (size, variant…)
- props the agent MUST NOT pass because the component derives them (Input `state`)
- props the agent SHOULD pass when a persistent user-owned state applies (ItemSideBar `state="active"` for the currently-selected page)

Audit identified 3 groupings:
- **Class A — auto-derived state prop** (Input, TextArea): doc examples currently show `state="..."` explicitly — misleading.
- **Class B — state variant in cva but NOT a prop** (Button, Checkbox, Radio, Switch, Collapsible, Tabulations, TabulationCells, Table): variants documentation can mislead agents.
- **Class C — state prop + no auto-derivation, persistent semantics** (ItemSideBar, ActionSheetItem, ListContent): legitimate consumer-owned state — keep the API, doc must classify it explicitly.

Goal: add a clear, mechanically-generated taxonomy to every component doc that removes ambiguity for both humans and agents, plus a global rule in `docs/agent-brief.md`. No source code refactor — evolution of the documentation only, driven by a JSDoc convention on each prop.

## Approach

Single source of truth is the component source (`.tsx`). Each prop in `export interface <Name>Props` gets a JSDoc tag `@propClass <class>` where class ∈ `required` | `optional` | `computed` | `persistent`. Default when absent: `optional` (with warning).

The documentalist agent parses these tags and produces a new **"Props usage"** section in each `docs/components/_index_<name>.md`, replacing the current API section in structure but retaining the existing props table with an added Class column. A companion **"Do / Don't"** block is emitted automatically for every component that has at least one `computed` or `persistent` prop.

Global rule added to `docs/agent-brief.md` § Composition rules via `scripts/build-agent-brief.ts`: "Never hardcode a prop classified `computed` — read the component's Props usage table before calling it."

## Prop classes

| Class | Meaning | Agent rule |
|-------|---------|------------|
| `required` | No sensible default; omitting it breaks the component. | Must pass. |
| `optional` | Has a default; override only when needed. | May pass. |
| `computed` | Auto-derived from HTML attributes or internal state. Passing freezes the UI. | Do NOT pass outside playground/demo. |
| `persistent` | Consumer-owned state with semantic meaning (current page, selected row). No sensible default. | Pass when relevant. |

## Files modified

- `src/components/*/`*.tsx — annotate each prop in every `*Props` interface with a `/** @propClass … */` JSDoc tag. No behavior change.
  - `input`, `text-area`: `state` → `computed`; `label`/`placeholder`/`description` → `optional`; `onChange`/`value` → `optional`.
  - `button`: no `state` prop (Class B); `variant`, `size` → `optional`; `children` → `required`.
  - `item-side-bar`, `list-content`: `state` → `persistent`; `feature` → `optional`.
  - `action-sheet-item`: `state` → `persistent`.
  - All others: annotate per existing semantics.
- `.claude/agents/documentalist.md` — extend DOCUMENT operation to:
  1. Parse JSDoc `@propClass` tags from the source's `export interface <Name>Props` block.
  2. Render a **"Props usage"** section with columns: Prop | Type | Default | Class | Description.
  3. If any prop is `computed` or `persistent`, emit a **"Do / Don't"** subsection directly under "Props usage".
  4. If a cva `state` axis exists but no `state` prop on the Props interface (Class B), insert a **"State variants"** explanatory note above the Variants table.
  5. Props without `@propClass` default to `optional` and a warning is added to the returned JSON report.
- `docs/components/_index_*.md` — all regenerated by the documentalist. Never hand-edited outside the KLP:NOTES block.
- `scripts/build-agent-brief.ts` — extend "Composition rules (hard)" with a new bullet about `computed` props. Append a "Computed & persistent props" appendix table, generated from `klp-components.json`.
- `klp-components.json` — documentalist adds a new field per component: `props: { <name>: { class, type, default } }`. Sole writer is documentalist.
- `CLAUDE.md` — one-line addition under "Component rules": "Every prop in a component's exported `Props` interface carries a `@propClass` JSDoc tag (`required` | `optional` | `computed` | `persistent`). The documentalist parses these."

## Doc template change

Rename "API" section to **"Props usage"** and add a Class column. Immediately after, when applicable, emit a Do / Don't block. Rendering for Input:

```markdown
## Props usage

| Prop | Type | Default | Class | Description |
|------|------|---------|-------|-------------|
| label | string | — | optional | Field label. |
| placeholder | string | — | optional | Placeholder text. |
| description | ReactNode | — | optional | Helper / error text. |
| state | `"default" \| "filled" \| …` | auto-derived | **computed** | Visual state override. Auto-derived from `disabled`, `aria-invalid`, focus, value. Do not pass. |
| disabled | boolean | false | optional | Native HTML disabled. Drives `state="disable"`. |
| aria-invalid | boolean | false | optional | Native ARIA. Drives `state="danger"`. |

### Do / Don't

✅ Do — let the component derive its own state:
\```tsx
<Input label="Email" placeholder="you@example.com" />
<Input label="Email" aria-invalid description="Invalid address" />
<Input label="Email" disabled />
\```

❌ Don't — do NOT freeze interactive state:
\```tsx
<Input state="default" />    {/* breaks focus/filled transitions */}
<Input state="filled" />     {/* freezes color in a real form */}
\```
```

Class B note (above the Variants table when no `state` prop exists on Props):

```markdown
> The `state` column below documents visual appearances driven by CSS pseudo-classes
> (`:hover`, `:focus`, `:disabled`) or the Radix `data-state` attribute. It is NOT a
> runtime prop — the component derives it automatically.
```

## Verification

1. Run documentalist with `operation: DOCUMENT, component: input`. Expect the new Props usage table with `state: computed` + a Do / Don't block.
2. Run documentalist with `operation: SYNC`. Expect every doc regenerated; `klp-components.json` gains the `props` map on every entry.
3. Run `pnpm run build:agent-brief && git diff docs/agent-brief.md`. Expect a new Composition rule + appendix table listing every `computed` and `persistent` prop DS-wide.
4. Spot-check Input doc section order: Anatomy → Variants → **Props usage → Do / Don't** → Tokens → Examples → Accessibility → …
5. Spot-check ItemSideBar doc: `state` marked `persistent` with a Do example showing `<ItemSideBar state="active" />`.
6. Spot-check Button doc: "State variants" note above the variants table, no Do / Don't block (no `computed` / `persistent` prop).
7. Grep `docs/components/_index_*.md` for `state="default"` / `state="filled"` in Examples. Expect zero hits on Input/TextArea.
8. Smoke-test the design agent: feed `docs/agent-brief.md` + `docs/components/_index_input.md`; confirm it emits `<Input label="…" />` rather than `<Input state="default" label="…" />`.

## Non-goals

- No source refactor of Class C components. Their `state` prop stays — it models persistent user-owned state, which is correct.
- No behavioral change in Input or TextArea. `state` keeps its override semantic; only examples stop showing it.
- No new cva variant, no new Tailwind utility, no new token.
- No migration of Class B components — they already route state through CSS / Radix.
