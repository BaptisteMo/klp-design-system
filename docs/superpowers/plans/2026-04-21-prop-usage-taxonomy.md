# Per-prop usage taxonomy — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Classify every component prop as `required` | `optional` | `computed` | `persistent` via JSDoc tags, surface that classification in every component doc page (new "Props usage" section + Do/Don't block + Class B explanatory note) and in `docs/agent-brief.md` (new composition rule + global appendix), so design agents stop hardcoding auto-derived props.

**Architecture:** Source-of-truth is the component `.tsx` — each prop in `export interface <Name>Props` carries a `/** @propClass <class> */` JSDoc tag. The documentalist parses tags on DOCUMENT/SYNC and renders a new template section. `scripts/build-agent-brief.ts` pulls per-component prop classifications from `klp-components.json` (populated by documentalist) and emits a global appendix.

**Tech Stack:** Markdown agent prompts, TypeScript (tsx scripts), JSDoc conventions, React 19 + TypeScript component sources, Node 22.

```

  lines.push('## Computed & persistent props (appendix)')
  lines.push('')
  lines.push('Derived from every component\'s `Props` interface `@propClass` tags. Computed = do NOT pass. Persistent = pass when relevant.')
  lines.push('')
  lines.push('| Component | Prop | Class | Description |')
  lines.push('|---|---|---|---|')
  for (const c of components) {
    const props = (c.props ?? {}) as Record<string, { class: string; type: string; default: string | null; description?: string }>
    for (const [name, meta] of Object.entries(props)) {
      if (meta.class === 'computed' || meta.class === 'persistent') {
        const desc = (meta.description ?? '').replace(/\|/g, '\\|').slice(0, 120)
        lines.push(`| \`${c.name}\` | \`${name}\` | **${meta.class}** | ${desc} |`)
      }
    }
  }
  lines.push('')
```

For each component in the table above, open its `.tsx` file and for every prop in the exported `interface *Props` block (or type alias), prepend a JSDoc comment of the form:

```tsx

   * Field label rendered above the input.

   */

```

If an existing JSDoc already exists above the prop, insert the `@propClass` tag as a new line at the end of the JSDoc (before the closing `*/`), without deleting the existing description. Example — existing JSDoc becomes:

```tsx

   * Explicit visual state override. When omitted, state is derived from `disabled`

   * @derivedFrom disabled, aria-invalid, focus, value

```

The `@derivedFrom` tag on `computed` props is a free-form comma-separated list of the HTML attributes / internal state that drive the derivation. The documentalist uses it to generate Do examples in Task 1 step 3 — include it for every `computed` prop.
